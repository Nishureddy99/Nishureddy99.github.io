3D VR Graph Visualization Tool

Research · Deployed

A cross-platform tool I built at OSU for HPC researchers to explore network datasets in immersive VR and on desktop. The biggest challenge was making it actually usable at scale - 10K+ nodes in a VR headset is a different problem than rendering the same data on a monitor. I had to rethink the entire rendering and loading pipeline to get there.

What I actually built
  • GPU instancing pipeline grouping nodes by mesh and material pair, issuing single DrawMeshInstanced calls per batch (1023 max) with per-instance color via MaterialPropertyBlock to avoid breaking the instancing group
  • Coroutine-driven batch loader that distributes node instantiation across frames with configurable batch size and delay - eliminated a 2–3 second VR freeze that was making the headset unusable at 10K+ nodes
  • Typed constraint-based filtering system that auto-detects parameter types (numeric, string, unit-with-SI-prefix, array) and normalizes SI units across the full pico-to-tera scale before comparison
  • Undo/redo system where every user action stores delegate pairs, and a dependency graph tracks which interactions share affected nodes so undo cascades correctly
  • Custom BiconnectedComponentsAlgorithm implementing Tarjan's articulation point detection via DFS, followed by flood-fill component membership and block-cut tree construction
  • OpenXR action-based input targeting both Meta Quest and desktop from shared application logic - same interaction code, different platform binders

NaVision: AR Accessibility System

Deployed · Quest 3

An AR assistant for visually impaired users that runs on Meta Quest 3. It streams passthrough camera frames to GPT-4o for real-time scene descriptions delivered through text-to-speech, with a parallel on-device ONNX pipeline so it can still detect objects without a network connection. The core design challenge was making inference reliable under mobile hardware constraints while keeping latency low enough that the audio feedback is actually useful.

What I actually built
  • Dual-backend inference with a shared interface contract - routes between on-device ONNX/YOLOv8 for fast detection and OpenAI GPT-4o for richer scene descriptions, with each backend independently testable and swappable
  • Quest 3 passthrough pipeline: camera intrinsics via PassthroughCameraUtils, frame capture as Texture2D, resize and TextureConverter.ToTensor, then layer-by-layer inference via ScheduleIterable yielding every 25 layers to keep the frame rate stable
  • 2D-to-3D spatial mapping: projects bounding box centers to world rays using camera intrinsics, then intersects with the Quest 3 scene mesh via EnvironmentRayCastSampleManager to get real-world distance and direction
  • Non-Maximum Suppression with IoU-based duplicate filtering at 0.6 threshold, sorted by confidence, with an indoor object allowlist to suppress irrelevant COCO classes
  • Voice-first interaction using the Wit.ai NLP SDK for wake word detection and natural language commands, with controller fallback for accessibility
  • Prioritized audio queue using Queue<AudioRequest> with rate limiting and new-detection diffing - announces the most important new objects without stepping on itself

// PROJECT_05

Smart Glasses AI Notes Apps

In Progress · Multi-platform

Building smart notes applications across two smart glasses platforms - Mentra Live and JL Allwinner. Both glasses have no display, so the entire interaction model is audio-first: the mic captures conversations, AI processes and stores them, and the user gets answers spoken back. The interesting engineering problem on each platform is different, which has been a useful forcing function for understanding how the underlying architectures actually compare.

Mentra Live - TypeScript / MentraOS SDK
  • Node.js app server connecting to MentraOS cloud relay over WebSocket - glasses → phone → cloud → server data path, with the phone acting as the BLE bridge and STT processor
  • Subscribes to onTranscription(isFinal) events rather than raw audio chunks - platform handles STT, server receives text, which keeps the architecture simpler and reduces compliance surface
  • Per-user conversation storage keyed by userId (stable across session restarts) with session-level sessionId - designed to survive BLE drops and phone lock screen kills without losing conversation context
  • Question intent detection on incoming transcription, followed by LLM retrieval call with accumulated transcript as context - answer delivered back through glasses speakers via TTS
  • Audio queue with rate limiting and new-detection diffing so spoken responses don't overlap or interrupt the user's ongoing speech
JL Allwinner — Kotlin / Android companion app
  • Android companion app connecting to JL Allwinner glasses over BLE GATT for commands and WiFi Direct for media transfer - two separate transport layers that have to be sequenced correctly
  • All hardware commands go through LargeDataHandler.glassesControl(byteArray) - photo trigger, audio start/stop, AI recognition — with responses parsed via GlassModelControlResponse.getWorkTypeIng() to track glasses state
  • Connection lifecycle gated on onServiceDiscovered() before any LargeDataHandler calls are allowed - a common silent failure point when commands are issued too early in the BLE handshake
  • WiFi Direct credential handshake: glasses push SSID and password to the phone over BLE, stored in DeviceManager, then GlassesControl.importAlbum() uses them to pull media files - the two transports coordinate through this shared state
  • PCM voice streaming via GlassesControl.receiveOpusData() with Opus/Speex native codec decoding - audio data decoded on the phone and forwarded to the AI pipeline for processing

Mixed-Methods User Study

Designed a counterbalanced within-subjects study comparing HMD-VR vs Desktop VR for graph analysis. Tasks were built from expert user feedback and the Brehmer–Munzner graph task taxonomy. I embedded NASA-TLX directly in the VR application so participants never had to remove the headset between task blocks - keeping the session flow intact.

n=14 participants · SUS · UEQ · In-app NASA-TLX · Think-aloud protocol · IRB-approved
⚡

GPU Performance Engineering

Tracked down and fixed a 2–3 second VR scene freeze at 10K+ nodes. The bottleneck wasn't the node count itself - it was synchronous VFX and shader initialization in Start(). I redesigned the loading pipeline to distribute instantiation across frames using coroutines, and added GPU instancing via Graphics.DrawMeshInstanced to cut draw calls.

60+ FPS sustained on Quest hardware · No scene freeze · Draw call batching via MaterialPropertyBlock
🔧

CI/CD and Build Pipeline

Containerized the Unity Linux build to run on Windows using Docker Desktop's WSL2 backend with WSLg graphics forwarding - this solved the OpenGL/GUI failures I kept hitting with a plain WSL-hosted Docker run. Multi-platform builds for Windows, Linux, and Quest shipped to customers on a weekly sprint cadence.

Docker · Azure DevOps · GitHub Actions · 60% reduction in deployment time
